A Courier Parcel Tracking System in C++

~5 min read·Programming Fundamentals project
C++ArraysData Handling

After building the University Management System with structures, I wanted to practice the same fundamentals a different way — so for this one I went with parallel arrays instead of a single struct, and built a console app that tracks parcels through a courier system from booking to delivery.

The data model

Instead of one array of a Parcel struct, each field gets its own array, all indexed by the same position:

  • trackingID[] — unique ID for each parcel
  • senderName[] / receiverName[]
  • city[] — destination city
  • weight[] — parcel weight in kg
  • status[] — Booked, In Transit, or Delivered

It's a more old-school way of organizing the data compared to structs, but it's a useful exercise in keeping several arrays in sync by index — every operation has to update all six arrays together or the record falls apart.

Finding a parcel

Every operation depends on being able to find a parcel by its tracking ID, so that logic lives in one shared function:

int findParcel(string id) {
  for (int i = 0; i < total; i++) {
    if (trackingID[i] == id) return i;
  }
  return -1;
}

It's a simple linear search, but reusing it everywhere — adding, searching, updating, deleting — meant I only had to get the "not found" handling right once.

Booking a parcel

Adding a parcel checks the array isn't full, then checks the tracking ID doesn't already exist using findParcel, before asking for the sender, receiver, destination city, and weight. Every new parcel starts life with the status "Booked" — nothing gets added in any other state.

Updating status

Parcels move through three stages: Booked → In Transit → Delivered. The update function looks the parcel up by tracking ID, shows the current status, and lets you pick the new one from a menu rather than typing free text — which avoids typos creating a status that doesn't match any of the three valid states.

Deleting without leaving gaps

This was the part I found most interesting. Deleting from the middle of an array normally means shifting every element after it back by one — which is slow and easy to get wrong across six parallel arrays at once. Instead, I used a swap-and-shrink approach: copy the last parcel's data into the slot being deleted, then shrink the count by one:

trackingID[index] = trackingID[total - 1];
senderName[index] = senderName[total - 1];
receiverName[index] = receiverName[total - 1];
city[index] = city[total - 1];
weight[index] = weight[total - 1];
status[index] = status[total - 1];
total--;

The order of parcels changes, but nothing else needs to move, and the operation stays fast no matter how many parcels are in the system.

The menu

Everything hangs off a single loop in main() with six options — add, display all, search, update status, delete, and exit — each calling its own dedicated function. Keeping main() to just the menu and the dispatch logic made it much easier to test each feature on its own.

What I'd improve next

Parallel arrays work, but they make it easy to update one array and forget another — a risk structs don't have. If I extended this project, moving to a struct (or a small class) per parcel would remove that whole category of bug, and adding persistence with file handling — like I did in the University Management System — would let the tracking data survive between runs instead of resetting every time the program closes.

What I learned

This project reinforced how much of "real" software is really just careful bookkeeping — keeping related data in sync, searching efficiently, and handling edge cases like deleting from the middle of a collection without breaking everything else. It's a smaller project than the University Management System, but it approaches the same kind of problem from a different angle, which is exactly why I built it.
