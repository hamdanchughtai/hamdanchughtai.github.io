How I Built My Portfolio

~4 min read·HTML, CSS, JavaScript
HTMLCSSJavaScript

This site started out as a single HTML file — a hero section, a couple of project cards, and my socials. That was enough to get something live, but it didn't leave much room to grow. So I rebuilt it into what's here now: a proper multi-page site with its own design language, animations, and even a small built-in assistant that answers questions about me.

The design direction

Since I spend most of my time writing code, I wanted the site itself to feel like a code editor rather than a generic template. That's where the whole look came from — the traffic-light dots at the top, the file-path breadcrumb, the monospace headings, and the terminal-style "$" prompts scattered through the page. Even the color palette is pulled straight from a dark syntax-highlighting theme: green for strings, blue for functions, amber for warnings.

Structure over a single page

The original version crammed everything — about, projects, contact — into one scrolling page. Once I wanted a real blog, a dedicated projects breakdown, and a uses page, that stopped making sense. I split it into separate pages (Home, About, Projects, Blog, Uses, Contact) that share one styles.css and one script.js file, so updating the design or the chat widget only has to happen in one place instead of six.

Little details that took the most time

The features that look simple usually took the longest to get right:

  • A scroll progress bar pinned to the top of the page
  • A typewriter effect that types out my name on the homepage
  • Sections and cards that fade and slide in as you scroll, staggered so they don't all animate at once
  • A mobile nav menu that collapses into a dropdown below a certain screen width

All of the animations respect the "reduce motion" setting too — if someone has that turned on in their OS, everything just appears instantly instead of animating.

The "Ask about me" widget

The green "?" button in the corner opens a small chat panel that answers common questions about me — my skills, my projects, how to get in touch — without needing a real AI model running in the background. A plain static site can't safely call an AI API directly (the API key would be exposed to anyone who views the page source), so instead it matches keywords in whatever's typed against a small knowledge base and replies instantly. No server, no cost, and it still feels responsive.

Why not just use a real LLM? A true AI-powered assistant needs a backend to hold an API key securely. That's a reasonable next step once the site needs it — for now, keyword matching covers the common questions well.

What's next

The plan going forward is to treat this less like a static resume and more like a running log — each new project gets written up here as it's finished, the same way this post and the ones on the University Management System and Smart Home Automation build came together.
